    微调 LoRA 训练

    07 - 模型微调与训练

    一、概述

    大模型微调（Fine-tuning）是将预训练基座模型适配到特定任务或领域的关键技术。2025-2026年，LoRA 及其变体已成为行业标准，RLHF/DPO 等对齐技术持续演进，GRPO 等新方法在推理模型中广泛应用。分布式训练工具链日趋成熟，Axolotl、Unsloth、LLaMA-Factory 等工具大幅降低了微调门槛。

    二、知识点详解

    1. 全量微调 vs PEFT

    全量微调（Full Fine-tuning）

    更新模型所有参数。效果上限高，但计算和存储成本巨大。

    • 显存需求：以 LLaMA-70B 为例，全量微调需 8x80G A100+
    • 存储需求：每个微调版本需存储完整模型权重（~140GB for 70B）
    • 适用场景：有充足算力、需要极致效果

    PEFT（Parameter-Efficient Fine-Tuning）

    只更新少量参数，大部分参数冻结。

    Adapter

    在 Transformer 层中插入小的瓶颈层（bottleneck）。

    Original:  LayerNorm -> Attention -> LayerNorm -> FFN
With Adapter: LayerNorm -> Attention -> LayerNorm -> FFN -> DownProject -> Activation -> UpProject -> Residual

    参数量约为原模型的 2-4%。

    Prefix Tuning

    在每层 Key 和 Value 前添加可学习的虚拟 Token（prefix）。

    LoRA ~1% 接近全量 不变

    2. LoRA / QLoRA 原理

    LoRA（Low-Rank Adaptation）

    核心思想：预训练权重矩阵的更新量 ΔW 具有低秩性，可分解为两个小矩阵的乘积。

    原始: h = Wx
LoRA: h = Wx + BAx  (其中 B∈R^{d×r}, A∈R^{r×k}, r << min(d,k))

    import torch.nn as nn

class LoRALayer(nn.Module):
    def __init__(self, in_dim, out_dim, rank=8, alpha=16):
        super().__init__()
        self.rank = rank
        self.alpha = alpha
        self.scaling = alpha / rank
        
        # 低秩矩阵 A（随机初始化）
        self.lora_A = nn.Parameter(torch.randn(in_dim, rank) * 0.01)
        # 低秩矩阵 B（全零初始化）
        self.lora_B = nn.Parameter(torch.zeros(rank, out_dim))
        
    def forward(self, x):
        # BA 是微调更新量
        return (x @ self.lora_A @ self.lora_B) * self.scaling
    • r（rank）：通常 8-64，控制可调参数量
    • alpha：缩放系数，控制微调幅度
    • 推理时可合并到原权重 W = W + BA，无额外延迟

    3. LoRA 变体（2024-2025）

    DoRA（Weight-Decomposed Low-Rank Adaptation）

    将预训练权重分解为幅度（magnitude）和方向（direction），只对方向部分做低秩更新。

    W = m * (V / ||V||)  # 幅度 + 单位方向
ΔW = m * (V + BA) / ||V + BA|| - m * V/||V||

    DoRA 比 LoRA 更接近全量微调的行为，在多项基准上提升 2-5%。

    PiSSA（Principal Singular Values and Singular Vectors Adaptation）

    对权重矩阵做 SVD 分解，只微调主奇异值对应的奇异向量。相比随机初始化的 LoRA，PiSSA 初始化更优，收敛更快。

    W = UΣV^T
仅更新前 r 个主奇异向量
ΔW = U_r · diag(Δσ) · V_r^T

    VeRA（Vector-based Random Adaptation）

    冻结 A 矩阵随机初始化不变，只训练 B 矩阵。参数量减半，效果略低于 LoRA 但更省显存。

    5. RLHF 完整流程

    Stage 1: SFT（Supervised Fine-Tuning）

    在高质量指令数据上监督微调，让模型学会遵循指令的格式。

    数据：人类标注的 (指令, 期望回答) 对
损失：Cross-Entropy Loss（标准语言模型损失）
目标：模型学会输出格式化和有帮助的回答

    Stage 2: Reward Model（奖励模型训练）

    训练一个独立的模型来打分，学习人类的偏好。

import torch.nn.functional as F

def reward_model_loss(reward_chosen, reward_rejected):
    """Pairwise ranking loss"""
    return -F.logsigmoid(reward_chosen - reward_rejected).mean()

    DPO（Direct Preference Optimization）

    def dpo_loss(policy_chosen_logps, policy_rejected_logps,
             ref_chosen_logps, ref_rejected_logps, beta=0.1):
    """DPO 损失函数"""
    # 对数概率差
    log_ratio_chosen = policy_chosen_logps - ref_chosen_logps
    log_ratio_rejected = policy_rejected_logps - ref_rejected_logps
    
    # 隐式奖励
    reward_chosen = beta * log_ratio_chosen
    reward_rejected = beta * log_ratio_rejected
    
    # Bradley-Terry loss
    loss = -F.logsigmoid(reward_chosen - reward_rejected).mean()
    return loss

    IPO（Identity Preference Optimization）

    改进了 DPO 的过拟合问题，简化损失函数。

    L_IPO = (log(π(y_w)/π_ref(y_w)) - log(π(y_l)/π_ref(y_l)) - 1/2τ)^2

    KTO（Kahneman-Tversky Optimization）

    使用"好/坏"评级的非配对数据，不需要严格偏好对。

    参考行为经济学中的前景理论
对好的回答鼓励靠近，对坏的回答惩罚远离

    SimPO（Simple Preference Optimization）

    移除 DPO 中的参考模型，使用平均对数概率作为隐式奖励，更简单高效。

    def simpo_loss(policy_chosen_scores, policy_rejected_scores, gamma=1.1):
    """SimPO 使用长度归一化的奖励"""
    # 直接使用模型输出的平均 logprob 作为奖励
    loss = -F.logsigmoid(
        (policy_chosen_scores - gamma) - policy_rejected_scores
    ).mean()
    return loss

    def orpo_loss(chosen_logps, rejected_logps, sft_logps, lambda_orpo=0.1):
    """ORPO = SFT loss + Odds Ratio loss"""
    sft_loss = -chosen_logps.mean()
    
    # Odds Ratio
    odds_chosen = torch.exp(chosen_logps)
    odds_rejected = torch.exp(rejected_logps)
    or_value = odds_chosen / (odds_rejected + 1e-8)
    or_loss = -F.logsigmoid(or_value).mean()
    
    return sft_loss + lambda_orpo * or_loss

    7. GRPO（Group Relative Policy Optimization）

    DeepSeek 提出的强化学习方法，在 DeepSeek-R1 等推理模型中广泛使用。

    核心思想：不从 Reward Model 获取绝对奖励，而是从一组生成样本中计算相对奖励。

    GRPO 流程：
1. 对每个 prompt，生成 G 个回答（group sampling）
2. 对每个回答计算奖励（可用规则奖励 + 模型奖励）
3. 在 group 内归一化奖励（减去均值，除以标准差）
4. 用归一化后的相对奖励更新策略
5. 引入 KL 散度约束，防止模型偏离基座

    def grpo_loss(policy_logprobs, rewards, ref_logprobs, beta=0.04):
    """GRPO 损失计算"""
    # 归一化奖励（group内）
    normalized_rewards = (rewards - rewards.mean()) / (rewards.std() + 1e-8)
    
    # 策略梯度损失
    ratio = torch.exp(policy_logprobs - ref_logprobs.detach())
    policy_loss = -(ratio * normalized_rewards).mean()
    
    # KL 惩罚
    kl = (torch.exp(ref_logprobs - policy_logprobs) - 
          (ref_logprobs - policy_logprobs) - 1).mean()
    
    return policy_loss + beta * kl

    优势：

    • 不需要训练 Reward Model（可使用规则奖励或 LLM-as-a-Judge）
    • 组内归一化提供了天然的基准
    • 在数学推理、代码生成等有明确答案的任务中效果优异

    8. 分布式训练基础

    Data Parallel（DP/ DDP）

    每个 GPU 持有完整模型副本，数据切分到不同 GPU。

    1

from torch.nn.parallel import DistributedDataParallel

    • 通信量：每个 batch 同步梯度
    • 瓶颈：大模型显存放不下

    Tensor Parallel（TP）

    将单个 Transformer 层的参数切分到多个 GPU。

    矩阵 W 的切分方式：
- Column-wise: W = [W1, W2, W3, W4]  (每GPU负责部分列)
- Row-wise: W = [W1; W2; W3; W4]     (每GPU负责部分行)
- 需要 all-reduce 通信

    Pipeline Parallel（PP）

    将模型的不同层分配到不同 GPU，形成流水线。

    GPU 0: Layers 1-8
GPU 1: Layers 9-16
GPU 2: Layers 17-24
GPU 3: Layers 25-32

输入 -> GPU0 -> GPU1 -> GPU2 -> GPU3 -> 输出

    存在 GPU 空闲（bubble）问题，常用 Micro-batching 缓解。

    FSDP（Fully Sharded Data Parallel）

    PyTorch 官方方案，将模型参数、梯度、优化器状态分片到所有 GPU。

    Arena Elo（Chatbot Arena）

    LMSYS 发布的众包评估平台，使用 Elo 评分系统。

    流程：
1. 用户提交问题
2. 两个模型匿名回答
3. 用户投票选择更好
4. 根据投票结果更新 Elo 分数

Elo 公式：
E_A = 1 / (1 + 10^((rating_B - rating_A) / 400))
new_rating_A = rating_A + K * (actual_score - E_A)

    优势：人类偏好直接、覆盖面广。缺点：噪音大、成本高。

    三、面试常见问题

    Q1：LoRA 的原理是什么？为什么有效？

    考察点：对低秩分解的数学理解和直觉

    参考答案：

    LoRA 基于核心假设：预训练模型在微调时，权重的更新量 ΔW 是低秩的。
即 ΔW 可以用两个低秩矩阵的乘积近似：ΔW = BA, B∈R^{d×r}, A∈R^{r×k}, r<<min(d,k)。

为什么有效：
1. 预训练模型已经学好了通用特征，微调只需要在低维子空间中调整
2. Aghajanyan et al. (2021) 证明了微调的语言任务通常具有低内在维度
3. 低秩约束天然起到正则化作用，减少过拟合

为什么推理无额外开销：训练完后可以将 BA 合并到 W 中：
W' = W + α/r * BA
推理时直接使用 W' 即可。

    Q2：Full Fine-tuning vs LoRA 如何选择？

    考察点：实际工程决策能力

    参考答案：

    选择 Full Fine-tuning 的情况：
1. 有充足算力（8+ A100）
2. 需要极限效果（领域差异极大）
3. 模型较小（<7B）
4. 需要权重量化或剪枝后的补偿

选择 LoRA 的情况：
1. 算力有限（单卡/消费级 GPU）
2. 需要快速迭代多个版本
3. 需要同时服务多个微调版本（动态切换 LoRA 权重）
4. 基座模型较大（>13B）
5. 微调数据量不大（<10K）

实际建议：先用 LoRA 快速验证，需要提升时再转为全量微调。

    Q3：RLHF 与 DPO 的区别是什么？

    考察点：对两种对齐方法的深入理解

    参考答案：

    RLHF（3-stage）：
1. SFT -> Reward Model -> PPO
2. 需要训练和维护独立的 Reward Model
3. 训练不稳定（PPO 超参数敏感）
4. 效果好上限高，适合大厂

DPO（2-stage）：
1. SFT -> DPO
2. 不需要 Reward Model，直接使用偏好对
3. 训练稳定，超参数少
4. 效果接近 RLHF，但可能受限于偏好数据质量

核心区别：DPO 将 RLHF 中的 Reward Model 阶段"隐式"融入了策略优化，
通过闭式解找到最优策略与奖励之间的关系，省去了显式的奖励训练。

    Q4：什么是 GRPO？和 PPO 的区别？

    考察点：2025年最新方法的理解

    参考答案：

    GRPO（Group Relative Policy Optimization）是 DeepSeek 提出的方法。

核心区别：
1. 无 Reward Model：GRPO 从组内样本计算相对奖励，PPO 依赖外部 Reward Model
2. 组内归一化：GRPO 对同 prompt 的 G 个回答的奖励做归一化，
   用相对奖励而非绝对奖励
3. 更简单：GRPO 不需要 Value Network（Critic），减少了模型规模和训练复杂度
4. 适合规则评估：在数学、编程等有明确答案的任务中，可直接用规则作为奖励

PPO 优势：对难以定义规则的任务（创意写作、对话）更通用
GRPO 优势：在推理类任务（数学、代码）中效果更好，训练更稳定

    Q5：微调数据准备需要注意什么？

    考察点：实际工程经验

    参考答案：

    1. 质量 > 数量：5K 高质量指令 > 50K 低质量
2. 去重：使用 MinHash 去除语义重复样本，避免模型过拟合常见模式
3. 多样性：覆盖对话、推理、创作、编程等不同任务类型
4. 难度分布：包含简单和困难样本，避免模型只学会处理简单问题
5. 格式一致：所有样本统一使用相同的对话模板（ChatML/ShareGPT）
6. 数据配比：不同任务按一定比例混合，避免某些任务主导训练
7. 质检流程：人工抽样 + LLM-as-a-Judge 双重质检
8. 多轮对话：确保 assistant 回答的 loss mask 正确，
   用户消息的 loss 设为 -100

## 一、操作系统概述

### 1. 定义与功能

操作系统是管理计算机硬件资源、提供用户接口、协调程序运行的底层系统软件，是计算机系统中最重要的系统软件。

**四大基本功能**：
- **处理器管理**：负责进程与线程的创建、撤销、调度，以及对 CPU 的分配与控制。
- **存储器管理**：内存分配与回收、地址映射（虚拟地址到物理地址）、内存保护、虚拟内存管理。
- **设备管理**：负责 I/O 设备的驱动、分配、缓冲和虚拟化（如 SPOOLing 技术）。
- **文件管理**：文件存储空间管理、目录组织、文件的读写与保护、权限控制。

### 2. 操作系统基本特征

**并发（Concurrency）**：宏观上在一段时间内可以同时运行多个程序。并发与并行的区别在于：并发是逻辑上的同时执行，并行是物理上的同时执行（需要多核或多处理器支持）。操作系统通过引入进程和线程来支持并发。

**共享（Sharing）**：系统中的资源可以被多个并发进程共同使用。两种共享方式：互斥共享（如打印机，同一时刻只允许一个进程访问）和同时共享（如磁盘文件，多个进程可同时读）。

**虚拟（Virtual）**：将一个物理实体转换为多个逻辑实体。时分复用技术（如 CPU 分时调度）让多个进程轮流占用处理器，每个进程感受到自己独占 CPU；空分复用技术（如虚拟内存）将物理内存抽象为更大的逻辑地址空间。

**异步（Asynchrony）**：进程以不可预知的速度向前推进，走走停停，并非一次性执行完毕。只要运行环境相同，操作系统必须保证多次运行同一程序得到相同结果。

### 3. 操作系统分类

| 类型 | 特点 | 典型代表 |
|:----:|:----:|:--------:|
| 批处理操作系统 | 无交互，作业成批进入内存 | 早期操作系统 |
| 分时操作系统 | 时间片轮转，交互性强 | UNIX、Linux |
| 实时操作系统 | 严格响应时限，必须满足截止时间 | μC/OS、VxWorks |
| 分布式操作系统 | 多机协同，资源共享，对用户透明 | Amoeba |
| 嵌入式操作系统 | 资源受限，专用性强 | FreeRTOS、嵌入式 Linux |

### 4. 中断与异常

**中断（Interrupt）**：CPU 暂停当前程序的执行，转而处理紧急事件，处理完成后返回原程序继续执行。中断是现代操作系统实现并发和响应的基础机制。

**中断分类**：
- **外中断（异步中断）**：来自 CPU 外部的事件，如 I/O 设备完成中断、时钟中断、控制台中断等。
- **内中断（异常，同步中断）**：来自 CPU 内部的事件，如除零、地址越界、缺页、非法操作码等。
- **陷入（Trap/系统调用）**：用户程序主动触发的中断，用于请求操作系统服务。

**中断响应过程**：关中断 → 保护断点（将 PC、PSW 等压栈）→ 查询中断向量表，转入中断服务程序 → 保存现场（通用寄存器入栈）→ 执行中断处理 → 恢复现场 → 开中断 → 返回断点。

### 5. 系统调用

**系统调用（System Call）**：用户程序请求操作系统内核服务的接口。用户程序通过陷入指令从用户态进入内核态，由操作系统完成内核态操作后返回用户态。

**用户态与内核态**：
- **用户态（目态）**：受限执行状态，不能执行特权指令（如 I/O 操作、中断开关）。
- **内核态（管态）**：可执行全部指令，包括特权指令。
- 切换时机仅三种：中断、异常、系统调用——这是用户态切换到内核态的唯一途径。

**常见系统调用**：进程控制（fork、exit、wait）、文件操作（open、read、write）、设备操作（ioctl、read）、信息维护（getpid、alarm）、通信（pipe、shmget、mmap）。

### 6. 宏内核与微内核

**宏内核（Monolithic Kernel）**：操作系统所有核心功能（进程管理、内存管理、文件系统、设备驱动等）均运行在内核态，模块之间可以直接调用，性能高。典型代表：Linux、Unix。

**微内核（Microkernel）**：只将最基本的功能放在内核态（如进程间通信、CPU 调度、基本内存管理），其他功能以服务形式运行在用户态。优点是内核小、扩展性好、安全性高；缺点是用户态/内核态频繁切换导致性能损失。典型代表：Minix、QNX。

## 二、进程与线程

### 1. 进程的概念

**进程（Process）**：程序的一次执行过程，是系统进行资源分配和调度的基本单位。

**进程与程序的区别**：程序是静态的指令集合，存储在磁盘上；进程是动态的执行过程，包含程序代码、数据和运行时上下文。同一程序可以对应多个进程（如同时打开多个浏览器窗口）。

**进程控制块（PCB, Process Control Block）**：操作系统管理进程的数据结构，是操作系统感知和管理进程的唯一依据。PCB 记录了：
- **标识信息**：进程 ID（PID）、父进程 ID、用户 ID
- **状态信息**：程序计数器（PC）、寄存器值、栈指针
- **控制信息**：进程状态（就绪/运行/阻塞）、优先级、内存指针、打开文件表、调度信息

### 2. 进程状态与转换

**五状态模型**：
1. **新建态（New）**：进程正在创建，尚未进入就绪队列。
2. **就绪态（Ready）**：进程已获得除 CPU 之外的所有需要的资源，等待调度程序分配 CPU。
3. **运行态（Running）**：进程正在 CPU 上执行。
4. **阻塞态（Blocked/Waiting）**：进程因等待某个事件（如 I/O 完成、信号量）而暂停执行。
5. **终止态（Terminated）**：进程执行结束，系统回收资源。

**状态转换规则**：
- 就绪态 → 运行态：调度程序选择该进程占用 CPU。
- 运行态 → 就绪态：时间片用完，或被更高优先级进程抢占。
- 运行态 → 阻塞态：进程发起 I/O 请求或等待资源。
- 阻塞态 → 就绪态：等待的事件完成。

### 3. 线程

**线程（Thread）**：CPU 调度的基本单位，是进程中的一条执行路径。一个进程可以包含多个线程，同一进程的线程共享进程资源。

**进程与线程的区别**：

| 对比维度 | 进程 | 线程 |
|:--------:|:----:|:----:|
| 基本定位 | 资源分配的基本单位 | CPU 调度的基本单位 |
| 地址空间 | 独立的虚拟地址空间 | 共享所属进程的地址空间 |
| 资源拥有 | 拥有独立的资源（文件表、信号处理等） | 只拥有少量执行现场（栈、寄存器、PC） |
| 切换开销 | 大（需切换地址空间、可能刷新 TLB） | 小（同一进程内切换无须切换地址空间） |
| 通信方式 | 需要 IPC（管道、消息队列、共享内存） | 可直接读写共享内存 |
| 故障影响 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程退出 |

**多线程模型**：
- **用户级线程**：由用户空间的线程库管理，内核不可见。切换快但一个线程阻塞会影响整个进程。
- **内核级线程**：由内核管理，可被调度到不同 CPU 上并行执行。切换开销大。
- **混合模型**：用户级线程映射到少量内核线程，综合两者优点。现代操作系统多采用此模型。

### 4. 进程通信（IPC）

进程间的通信方式，按数据交换量和使用频率排序：

**管道（Pipe）**：半双工通信，数据在内存中流动，常用于父子进程之间。命名管道（FIFO）可用于无亲缘关系的进程。

**消息队列（Message Queue）**：消息的链表，存放在内核中。进程通过系统调用发送和接收消息，有明确的边界。

**共享内存（Shared Memory）**：多个进程映射同一块物理内存，是最快的 IPC 方式。但需要信号量等机制同步访问。

**信号量（Semaphore）**：用于进程间同步和互斥，也可作为通信手段。

**套接字（Socket）**：可用于不同机器之间的进程通信，是最通用的 IPC 方式。

### 5. 调度算法

**调度层次**：
- **长程调度（作业调度）**：将外存中的作业调入内存，创建进程。发生频率低。
- **中程调度（交换调度）**：内存不足时将进程挂起，换出到外存。发生频率中等。
- **短程调度（进程调度）**：从就绪队列中选择一个进程分配 CPU。发生频率最高。

**主要调度算法**：

| 算法 | 类型 | 特点 | 适用场景 |
|:----:|:----:|:----:|:--------:|
| 先来先服务（FCFS） | 非抢占 | 按请求顺序调度，公平但平均等待时间长，对短作业不利 | 批处理系统 |
| 短作业优先（SJF） | 非抢占 | 选估计运行时间最短的作业，最小平均周转时间，长作业可能饿死 | 批处理系统 |
| 最短剩余时间优先（SRTN） | 抢占 | SJF 的抢占版本，新作业剩余时间更短则抢占当前作业 | 批处理系统 |
| 时间片轮转（RR） | 抢占 | 每个进程轮流执行一个时间片。时间片太大退化为 FCFS，太小切换开销过大 | 分时系统 |
| 优先级调度 | 均可 | 按优先级调度，低优先级可能饿死（可通过老化提高等待进程的优先级） | 通用系统 |
| 多级反馈队列 | 抢占 | 多个队列，各队列时间片不同，进程可在队列间升降级。综合兼顾交互型和计算型任务 | 现代通用操作系统 |

**调度指标**：
- **周转时间** = 完成时间 − 到达时间
- **平均周转时间** = 总周转时间 / 作业数
- **等待时间** = 周转时间 − 服务时间
- **响应时间** = 首次被调度时间 − 到达时间

## 三、进程同步与互斥

### 1. 临界区问题

**临界资源**：一次只允许一个进程使用的资源（如打印机、共享变量、文件等）。

**临界区（Critical Section）**：访问临界资源的代码段。对临界区的访问必须满足三条原则：
- **互斥**：任意时刻最多只有一个进程在临界区内执行。
- **前进**：当临界区空闲时，若多个进程同时请求进入，应能在有限时间内选择一个进入。
- **有限等待**：进程发出进入请求后，必须在有限时间内获准进入。

### 2. 信号量与 PV 操作

**信号量（Semaphore）**：一个整型变量 S，用于表示可用资源的数量。S > 0 表示有 S 个资源可用；S ≤ 0 表示有 |S| 个进程在等待该资源。

**PV 操作**（均为原子操作）：
- **P(S)**（wait/down）：S = S - 1；若 S < 0，则将当前进程阻塞并放入等待队列。
- **V(S)**（signal/up）：S = S + 1；若 S ≤ 0，则从等待队列中唤醒一个进程。

当信号量的取值只能为 0 或 1 时，称为**互斥量（Mutex）**，用于实现临界区的互斥访问。

**PV 操作使用的关键原则**：P 操作测试资源是否可用（资源不够时阻塞），V 操作释放资源（可能有等待者被唤醒）。P 和 V 必须成对出现且不可分割。

### 3. 管程（Monitor）

管程是一种高级同步机制，它将共享变量和对这些变量的操作封装在一个模块中。管程的特性是：**同一时刻只能有一个进程在管程内执行**。管程使用条件变量（Condition Variable）和 wait()/signal() 操作实现同步，比直接使用信号量更加结构化、不易出错。

### 4. 经典同步问题

**① 生产者-消费者问题**：
- 问题描述：生产者向有限缓冲区中放入数据，消费者从缓冲区取走数据。
- 同步要求：缓冲区满时生产者不能放，缓冲区空时消费者不能取；生产者和消费者不能同时访问缓冲区。
- 解决方案：使用三个信号量——`mutex`（互斥访问缓冲区）、`empty`（空槽数量）、`full`（满槽数量）。生产者先 P(empty) 再 P(mutex)，消费者先 P(full) 再 P(mutex)。注意 P(empty/full) 必须在 P(mutex) 之前，否则可能发生死锁。

**② 读者-写者问题**：
- 问题描述：允许多个读者同时读数据，但不允许读者和写者同时操作，也不允许多个写者同时写。
- 读者优先方案：第一个读者进入时锁定写者，最后一个读者离开时解锁写者；用 `read_count` 计数读者数量，用 `count_mutex` 保护 `read_count`，用 `data_mutex` 控制读者和写者的互斥。
- 可能存在写者饥饿问题，可通过增加 `write_count` 和对应的互斥信号量实现写者优先或公平策略。

**③ 哲学家进餐问题**：
- 问题描述：五位哲学家围坐圆桌，每人间放一根筷子。哲学家思考时不需要筷子，吃饭时必须同时拿起左右两根筷子。
- 风险：如果所有哲学家同时拿起左边的筷子，将全部等待右边的筷子，导致死锁。
- 解决方案：① 最多允许四位哲学家同时就餐；② 奇数号哲学家先拿左边再拿右边，偶数号先拿右边再拿左边；③ 用互斥信号量保护拿筷子的操作（一次性拿两把筷子）。

**④ 吸烟者问题**：三个吸烟者各有不同缺失的材料，供应者随机提供两种材料，用信号量同步协调。

## 四、死锁

### 1. 死锁的定义与必要条件

**死锁（Deadlock）**：多个进程因互相等待对方持有的资源而陷入无限等待的状态。

**四个必要条件**（必须同时满足）：
1. **互斥**：资源一次只能分配给一个进程使用。
2. **占有并等待**（持有并等待）：进程已持有部分资源，同时等待其他进程释放资源。
3. **非抢占**：已分配给进程的资源不能被强制剥夺，只能由进程主动释放。
4. **循环等待**：存在一条进程资源的循环等待链，链中每个进程都在等待下一个进程占有的资源。

**资源分配图**：用有向图描述进程和资源之间的分配与请求关系。当图中存在环时，可能发生死锁（每类资源只有一个实例时，有环即死锁；有多个实例时需要进一步判断）。

### 2. 死锁处理策略

| 策略 | 方法 | 效果 | 实际应用 |
|:----:|:----:|:----:|:--------:|
| 预防 | 破坏四个条件之一 | 降低资源利用率 | 常见于应用代码 |
| 避免 | 分配前判断是否安全 | 需预知最大需求 | 银行家算法 |
| 检测 | 允许死锁发生，定期检测 | 检测有成本 | 数据库系统常见 |
| 恢复 | 检测到死锁后解除 | 可能丢失工作 | 终止进程或回滚事务 |

### 3. 死锁预防

- **破坏互斥**：将资源改为共享（如只读文件、假脱机打印机技术）。但大多数资源本质互斥，不可行。
- **破坏占有并等待**：进程在开始执行前一次申请全部所需资源。资源利用率低，可能饥饿。
- **破坏非抢占**：进程申请新资源被拒时，必须释放已占有的资源。实现复杂。
- **破坏循环等待**：为所有资源编号，进程只能按编号递增顺序申请资源。最常用的预防手段。

### 4. 死锁避免——银行家算法

**银行家算法（Banker's Algorithm）**：在资源分配前模拟分配，判断分配后的系统状态是否安全。只有安全状态才允许分配。

**数据结构**：
- Available[m]：每种资源的可用数量
- Max[n][m]：每个进程对每种资源的最大需求
- Allocation[n][m]：每个进程已分配的资源数
- Need[n][m] = Max − Allocation：每个进程还需要的资源数

**安全状态检测步骤**：
1. 找一个 Need 向量 ≤ Available 的未完成进程 Pi。
2. 假设 Pi 完成并释放资源：Available += Allocation[i]。
3. 标记 Pi 为已完成，重复步骤 1。
4. 若所有进程均能标记完成，则状态安全；否则不安全。

**核心思想**：不阻止死锁的发生条件，而是确保系统始终处于可以避免死锁的安全状态。

### 5. 死锁检测与恢复

**检测**：定期检测资源分配图中是否存在循环等待，或使用类似银行家算法的矩阵检测法判断是否有进程无限等待。

**恢复方式**：
- **终止进程**：终止所有参与死锁的进程，或逐个终止直到死锁解除。
- **抢占资源**：从某些进程中强制剥夺资源给其他进程，需考虑回滚成本。
- **回滚**：将进程回滚到安全状态点，重新执行。

## 五、内存管理

### 1. 连续内存分配

| 分配方式 | 特点 | 碎片类型 |
|:--------:|:----:|:--------:|
| 单一连续分配 | 一道程序独占全部内存 | 无 |
| 固定分区分配 | 内存划分成固定大小的分区 | 内部碎片 |
| 动态分区分配 | 按需分配分区，分区大小可变 | 外部碎片 |

**动态分区分配算法**：

| 算法 | 选择策略 | 特点 |
|:----:|:--------:|:----:|
| 首次适应（FF） | 从低地址查找第一个合适分区 | 简单，低地址易产生小碎片 |
| 最佳适应（BF） | 查找最小的合适分区 | 碎片最小但需排序，产生大量小碎片 |
| 最坏适应（WF） | 查找最大的合适分区 | 减少小碎片，但大分区快速被分割 |
| 循环适应（NF） | 从上次分配位置继续查找 | 使内存分配更均匀 |

**外部碎片**的解决方式：**紧凑（Compaction）**——移动已分配分区使之连续，但成本较高。

### 2. 分页存储管理

**基本思想**：将进程的逻辑地址空间划分为固定大小的"页"（Page），物理内存划分为等大的"页框"/"帧"（Frame）。通过页表（Page Table）建立页到页框的映射。

**地址转换**：
- 逻辑地址 = 页号 P + 页内偏移 W
- 物理地址 = 页框号 f × 页面大小 + W（其中 f = 页表[P]）

**页表项结构**：页框号 + 标志位（有效位/存在位表示页面是否在内存中；修改位表示页面是否被写过；访问位表示页面是否被访问过；权限位控制读写执行等）。

**快表（TLB, Translation Lookaside Buffer）**：
- CPU 内部的关联存储器，缓存最近使用的页表项。
- TLB 命中时可直接得到物理地址，避免内存访问查页表。
- TLB 未命中时需查内存页表，然后将结果加入 TLB。

**多级页表**：
- 单级页表空间开销大：32位系统页表需约 4MB 空间。
- 多级页表按需创建下级页表，节省空间。x86-64 通常使用 4 级分页。
- 多级页表增加了地址转换的访问次数，这正是 TLB 发挥重要作用的场景。

**分页的优点**：无外部碎片（页可离散存放）；方便实现虚拟内存（只将需要的页调入内存）。

### 3. 分段存储管理

**基本思想**：按程序的逻辑结构将地址空间划分为段（如代码段、数据段、堆栈段），每段有独立的基址和界限。

**逻辑地址 = 段号 S + 段内偏移 W**：地址转换时查段表获得基址，将基址 + 偏移即可。偏移不能超过段界限。

**分段的特点**：贴近程序员的逻辑视角，便于数据和代码的共享与保护。但段长不固定，易产生外部碎片。

### 4. 段页式存储管理

先按逻辑分段，每个段内再分页。一个逻辑地址需要三次转换：段号 → 段内页号 → 页内偏移。兼顾了分段的逻辑清晰和分页的离散分配优势，但地址转换过程更复杂。

### 5. 分页与分段的对比

| 对比维度 | 分页 | 分段 |
|:--------:|:----:|:----:|
| 划分依据 | 固定大小（系统决定） | 程序逻辑结构（程序员决定） |
| 地址空间 | 一维 | 二维 |
| 大小是否可变 | 固定 | 可变（动态增长） |
| 主要目的 | 实现虚拟内存，消除外部碎片 | 便于共享和保护 |
| 对程序员透明 | 透明 | 不透明（需显式分段） |

## 六、虚拟内存

### 1. 基本概念

**虚拟内存（Virtual Memory）**：使程序可以使用比物理内存更大的地址空间的技术。其核心原理是**局部性原理**：
- **时间局部性**：刚访问过的地址很可能很快再次被访问。
- **空间局部性**：刚访问过的地址附近很可能也将被访问。

**按需调页（Demand Paging）**：只将进程当前需要的页面装入内存，其他页面暂存在外存（磁盘）中。当进程访问的页面不在内存时触发**缺页中断**（Page Fault），操作系统将该页从外存调入内存。

**缺页中断与普通中断的区别**：
- 缺页中断在指令执行期间触发（指令可能访问多个内存地址）。
- 缺页中断处理后需重新执行被中断的指令（因为该指令所需页面刚被调入）。

### 2. 页面置换算法

当物理内存已满且需要调入新页时，必须选择一个页面换出。页面置换算法的目标是使缺页率最低。

| 算法 | 置换策略 | 特点 |
|:----:|:--------:|:----:|
| OPT（最佳置换） | 置换未来最长时间不会访问的页 | 理论最优，不可实现 |
| FIFO（先进先出） | 置换最早进入内存的页 | 简单，可能发生 Belady 异常 |
| LRU（最近最久未使用） | 置换最长时间未被访问的页 | 性能好，实现成本高 |
| NRU（最近未使用） | 按 R（访问位）和 M（修改位）分四类，选编号最小类随机置换 | 近似 LRU，成本低 |
| 第二次机会 | 检查最老页的 R 位：1 则清 0 放回链表尾，0 则置换 | FIFO 的改进 |
| Clock（时钟） | 环形链表 + 指针遍历，R=1 清 0 继续，R=0 置换 | 第二次机会的环形实现 |

**Belady 异常**：使用 FIFO 算法时，分配的物理块增多反而导致缺页率上升。其他算法（LRU、OPT）不存在此异常。

**抖动（Thrashing）**：进程频繁缺页，导致大量时间花在页面换入换出上，CPU 利用率急剧下降。根本原因是进程的工作集（活跃使用的页面集合）大于物理内存容量。

### 3. 页面分配策略

- **固定分配局部置换**：为每个进程分配固定数量的物理块，缺页时仅置换自己的页面。
- **可变分配全局置换**：物理块数可变，可从全局空闲块中分配新块。
- **可变分配局部置换**：缺页时系统可分配新块，但置换时只置换自己的页面。

## 七、文件系统

### 1. 文件的逻辑结构

- **顺序文件**：记录按顺序存放（如磁带文件、日志文件）。
- **索引文件**：索引表 + 数据文件，支持快速随机访问。
- **索引顺序文件**：索引表指向一组顺序记录，综合索引和顺序访问的优点。

### 2. 文件物理结构（存储方式）

| 方式 | 特点 | 适用场景 |
|:----:|:----:|:--------:|
| 连续分配 | 文件在磁盘上连续存放 | 顺序访问快，便于扩展 |
| 链接分配 | 每个数据块包含指向下一块的指针 | 顺序访问好，随机访问差 |
| 索引分配 | 文件索引块存放所有数据块指针 | 随机访问好（如 inode） |

**FAT（文件分配表）**：链式分配的一种实现。整个文件系统有一张 FAT 表，记录每个块的下一块指针。根目录区 + 数据区 + FAT 表的结构。

**Unix/Linux inode 结构**：每个文件有一个索引节点（inode），存储文件元数据（权限、大小、时间）以及数据块指针。包含直接块指针、一级间接指针、二级间接指针、三级间接指针，可支持极大文件。

### 3. 目录结构

- **单级目录**：所有文件在同一目录下，简单但不能重名。
- **两级目录**：主目录（MFD）→ 用户目录（UFD），不同用户文件可重名。
- **树形目录**（现代操作系统）：路径名区分文件，支持目录嵌套，是使用最广泛的目录结构。

**空闲空间管理**：位图（Bitmap，用 0/1 标记每个块是否空闲）、空闲链表、成组链接（Unix 的成组链接法）。

### 4. 文件系统的可靠性

**硬链接**：多个文件名指向同一个 inode，文件的 inode 引用计数减少后文件才真正删除。

**软链接（符号链接）**：一个文件包含指向另一个文件的路径，如果目标文件被删除则链接失效。

## 八、I/O 管理

### 1. I/O 控制方式

| 方式 | CPU 参与程度 | 数据传输单位 | 特点 |
|:----:|:------------:|:------------:|:----:|
| 程序查询方式（轮询）| 持续查询设备状态 | 字/字节 | CPU 效率极低，设备被动等待 |
| 中断驱动方式 | 每次传送后中断通知 CPU | 字/字节 | 释放 CPU 等待时间，但中断处理有开销 |
| DMA 方式 | 初始化 + 完成后中断 | 数据块 | DMA 控制器直接搬运数据，不占用 CPU |
| 通道方式 | 更智能的 DMA | 数据块 | 专用 I/O 处理器，可执行通道程序 |

**DMA 的工作流程**：
1. CPU 设置 DMA 控制器：传送方向、内存首地址、传送字节数。
2. DMA 控制器控制总线进行数据传输（周期窃取方式最常用）。
3. 传输完成后 DMA 发起中断通知 CPU。

### 2. 缓冲技术

解决 CPU 与 I/O 设备间速度不匹配的问题：

- **单缓冲**：一次处理一块数据，处理与输入可部分重叠。
- **双缓冲**：交替使用两个缓冲区，可连续处理数据。
- **循环缓冲**：多个缓冲区组成环形，生产与消费并行。

### 3. SPOOLing 技术

**原理**：用磁盘的输入/输出井模拟独占设备，使独占设备变为共享设备。

**组成**：输入进程 + 输出进程 + 输入井 + 输出井。典型应用是打印机共享——多个程序同时"打印"时先写磁盘输出井，由 SPOOLing 系统逐个输出到物理打印机。

### 4. 磁盘调度算法

磁盘访问时间由三部分组成：寻道时间（磁头移动到目标磁道，占主要部分）+ 旋转延迟（旋转到目标扇区）+ 传输时间（实际数据读写）。寻道时间是主要优化目标。

| 算法 | 策略 | 特点 |
|:----:|:----:|:----:|
| FCFS（先来先服务） | 按请求顺序服务 | 公平但平均寻道时间长 |
| SSTF（最短寻道时间优先） | 选择离当前磁头最近的请求 | 平均寻道时间短，磁道两端的请求可能饿死 |
| SCAN（电梯算法） | 沿一个方向移动直到该方向无请求，再反向 | 解决饥饿，但两端请求等待时间长 |
| C-SCAN（循环扫描） | 单向移动到尽头后立即返回起点，重新开始 | 各磁道等待时间均匀 |

### 5. 设备独立性

用户程序使用逻辑设备名（如 `/dev/lp0`），操作系统通过逻辑设备表（LUT）将其映射为物理设备。引入逻辑设备名后，用户程序与具体物理设备解耦，更换设备时只需修改映射关系。

## 九、链接

### 1. 编译系统

从源代码到可执行文件的完整过程：

- **预处理**：处理以 `#` 开头的预处理指令（宏展开、文件包含、条件编译）。
- **编译**：将预处理后的源文件翻译成汇编代码。
- **汇编**：将汇编代码翻译成可重定位目标文件（机器指令 + 符号信息）。
- **链接**：将多个可重定位目标文件和库文件合并为可执行文件。

### 2. 静态链接

静态链接器将一组可重定位目标文件组合为可执行目标文件。主要完成两件事：
- **符号解析**：将每个符号引用与对应的符号定义关联起来。
- **重定位**：为每个符号分配内存地址，并修改所有对它的引用。

**静态库的缺点**：库更新后所有引用程序需重新链接；多个程序引用相同库时，库代码被复制到每个可执行文件中，浪费磁盘和内存。

### 3. 动态链接

**共享库（动态链接库）**：在 Linux 中以 `.so` 后缀存在，Windows 中为 DLL。动态链接有以下特点：
- 库在文件系统中只有一个副本，不在每个可执行文件中重复。
- 库的代码段在内存中可被多个进程共享（一个副本映射到多个进程的地址空间）。
- 程序运行时才加载库，更新库时不需要重新链接程序。

**动态链接的缺点**：运行时需额外的加载和链接时间；可能引发"DLL Hell"（版本冲突问题）。

## 十、I/O 多路复用

I/O 多路复用是指用一个进程/线程同时监视多个 I/O 事件（如网络连接），任意一个事件就绪时通知应用程序处理。这是实现高并发网络服务器的关键技术。

**三种主要实现机制**：

**select**：
- 将需要监视的文件描述符集合传递给内核，内核返回就绪的描述符。
- 最大监视数量受限（通常 1024）。
- 每次调用都需要将整个 fd 集合从用户态复制到内核态，效率随 fd 数量增加而下降。
- 返回后需要遍历所有 fd 来确定哪些就绪。

**poll**：
- 使用 pollfd 结构体数组，突破 select 的 1024 限制。
- 但仍需遍历整个数组，且每次调用需从用户态复制 fd 集合。
- 性能随 fd 数量增加而线性下降。

**epoll**（Linux 特有）：
- **事件驱动**：只返回就绪的事件，无需遍历所有 fd。
- **mmap 共享内存**：避免用户态和内核态之间的数据传输。
- **支持水平触发和边沿触发**，后者只通知一次事件状态变化。
- 在大规模并发连接场景下性能远优于 select 和 poll。

**应用场景**：Nginx、Node.js、Redis 等高性能网络服务均使用 epoll（Linux）或类似机制（如 kqueue、IOCP）。

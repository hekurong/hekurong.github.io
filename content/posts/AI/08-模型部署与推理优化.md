# 模型部署与推理优化

## 概述

模型部署与推理优化是AI工程化落地的核心环节。随着大语言模型（LLM）参数规模从数十亿增长至数千亿，将模型高效、低成本地部署到生产环境面临巨大挑战。本章涵盖模型导出、量化、推理引擎、分布式推理、部署架构等全链路知识，帮助面试者系统掌握从模型训练完成到线上高并发服务的完整技术栈。

## 知识点详解

### 一、模型导出（Model Export）

模型导出是将训练框架（PyTorch、TensorFlow、JAX）中的模型转换为推理优化格式的过程。

#### 1. ONNX（Open Neural Network Exchange）

ONNX 是微软和Meta推出的开放神经网络交换格式，作为不同框架间的中间表示。

```python
import torch
import torch.onnx

# PyTorch模型导出为ONNX
model = YourModel().eval()
dummy_input = torch.randn(1, 3, 224, 224)

torch.onnx.export(
    model,
    dummy_input,
    "model.onnx",
    opset_version=17,
    input_names=["input"],
    output_names=["output"],
    dynamic_axes={"input": {0: "batch_size"}, "output": {0: "batch_size"}}
)
```

关键参数：
- opset_version：算子集版本，越高支持更多算子
- dynamic_axes：动态维度声明，支持可变batch size
- do_constant_folding：常量折叠优化

#### 2. TensorRT（NVIDIA）

NVIDIA TensorRT 是面向GPU的高性能推理优化器，支持FP16、INT8、INT4量化。

```python
import tensorrt as trt

# ONNX -> TensorRT Engine
logger = trt.Logger(trt.Logger.WARNING)
builder = trt.Builder(logger)
network = builder.create_network(1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH))
parser = trt.OnnxParser(network, logger)

with open("model.onnx", "rb") as f:
    parser.parse(f.read())

config = builder.create_builder_config()
config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 1 << 30)  # 1GB
config.set_flag(trt.BuilderFlag.FP16)

serialized_engine = builder.build_serialized_network(network, config)
with open("model.engine", "wb") as f:
    f.write(serialized_engine)
```

#### 3. OpenVINO（Intel）

Intel OpenVINO 针对Intel CPU、GPU、NPU优化，适合边缘部署。

```bash
# ONNX -> OpenVINO IR

```

#### 4. CoreML（Apple）

Apple CoreML 用于iOS/macOS端侧推理。

### 二、量化技术（Quantization）

量化是将模型权重和激活值从高精度（FP32）映射到低精度（INT8、INT4）的技术，显著减少内存占用和推理延迟。

#### 1. PTQ（Post-Training Quantization）

训练后量化，无需重新训练，使用校准数据集确定量化参数。

```python
# PyTorch静态量化示例
import torch
import torch.quantization as quant

model = YourModel().eval()
model.qconfig = quant.get_default_qconfig("fbgemm")
quant.prepare(model, inplace=True)

# 校准：用代表性数据跑几次forward
for data in calibration_dataloader:
    model(data)

quant.convert(model, inplace=True)
# 模型现在以INT8运行
```

#### 2. QAT（Quantization-Aware Training）

量化感知训练，在训练过程中模拟量化误差，精度优于PTQ。

```python
# PyTorch QAT
model = YourModel().train()
model.qconfig = quant.get_default_qat_qconfig("fbgemm")
quant.prepare_qat(model, inplace=True)

# 正常训练
for epoch in range(num_epochs):
    for data, target in train_loader:
        output = model(data)
        loss = criterion(output, target)
        loss.backward()
        optimizer.step()

# 转换回INT8推理模式
quant.convert(model.eval(), inplace=True)
```

#### 3. INT4 / INT8 量化

| 精度 | 内存节省 | 推理加速 | 精度损失 |
|------|---------|---------|---------|
| FP32 | 1x (基线) | 1x | 0% |
| FP16 | 2x | ~1.5x | 极小 |
| INT8 | 4x | ~2-4x | 较小 |
| INT4 | 8x | ~3-6x | 较大(需特殊处理) |

#### 4. LLM专用量化格式

**GGUF（GPT-Generated Unified Format）**
- llama.cpp 生态使用
- 支持 2-8 bit 混合量化
- 适合CPU/边缘设备推理

```bash
# 使用llama.cpp进行量化
./quantize models/llama-7b.gguf Q4_K_M
```

**GPTQ（GPT Post-Training Quantization）**
- 基于Optimal Brain Quantization（OBQ）算法
- 逐层量化，考虑Hessian矩阵
- 支持4bit量化且精度优秀

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
from auto_gptq import AutoGPTQForCausalLM

model = AutoGPTQForCausalLM.from_quantized(
    "model_name",
    use_triton=True,
    quantize_config=None,
    device="cuda:0"
)
```

**AWQ（Activation-aware Weight Quantization）**
- 基于激活值分布感知的量化
- 保护"重要"权重通道（salient channels）
- 速度优于GPTQ，精度相当

```python
from awq import AutoAWQForCausalLM

model = AutoAWQForCausalLM.from_pretrained("model_name")
model.quantize(
    tokenizer,
    quant_config={"zero_point": True, "q_group_size": 128, "w_bit": 4}
)
model.save_pretrained("quantized_model")
```

### 四、推理优化技术

#### 1. PagedAttention

vLLM的核心创新，解决KV Cache的内存管理问题。

传统方案：
- KV Cache预先分配固定大小 -> 内存浪费（internal fragmentation）
- 无法共享请求间的KV Cache

PagedAttention：
- 将KV Cache分页管理（类似OS虚拟内存）
- 按需分配页面，消除碎片
- 支持页面级共享（Copy-on-Write）
- 内存利用率从~30%提升至~95%

#### 2. Flash Attention

I/O-aware的精确注意力计算算法，减少HBM访问。

核心思想：
- 将Q、K、V分块存入SRAM（片上高速缓存）
- 在SRAM内完成注意力计算
- 减少HBM读写次数

版本演进：
- FlashAttention 1：2x加速，精确注意力
- FlashAttention 2：进一步优化parallelism，~2x再加速
- FlashAttention 3（Hopper GPU）：利用FP8，~1.5x再加速

```python
# 在模型中启用Flash Attention
import torch
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B",
    torch_dtype=torch.bfloat16,
    attn_implementation="flash_attention_2",  # 启用Flash Attention
    device_map="auto"
)
```

#### 3. Continuous Batching（连续批处理）

传统批处理：等当前批次全部完成后才处理新请求。

连续批处理：
- 每个请求独立管理生成进度
- 请求完成后立即腾出slot给新请求
- 无需等待整个批次完成
- GPU利用率大幅提升，延迟降低

```text
传统批处理:
|---请求A---请求A---请求A---| |---请求B---请求B---| |---请求C---请求C---请求C---|
                              (GPU空闲)           (GPU空闲)

连续批处理:
|---请求A---请求A---请求C---请求B---请求A---请求C---请求B---请求C---|
              (请求A完成，新请求C加入)      (请求B完成，新请求...)
```

#### 4. Prefix Caching（前缀缓存）

缓存公共前缀（如系统提示词、对话历史）的KV Cache。

实现方式：
- SGLang: RadixAttention（基于Radix Tree的KV缓存）
- vLLM: Automatic Prefix Caching
- TGI: Prefix Caching

```python
# vLLM启用Prefix Caching
from vllm import LLM

llm = LLM(
    model="meta-llama/Llama-3.1-8B",
    enable_prefix_caching=True  # 启用自动前缀缓存
)
```

### 五、分布式推理

#### 1. Tensor Parallel（张量并行）

将单个层的权重矩阵切分到多个GPU上并行计算。

适用于：单节点多GPU场景

```text
输入
  |
[Layer]  ---  GPU0: 权重前半部分
  |       ---  GPU1: 权重后半部分
  |
  输出(需all-reduce合并)
```

- 通信量：每层需要all-reduce
- 延迟：低（通信量小）
- 限制：GPU需在同一个节点（NVLink）

#### 2. Pipeline Parallel（流水线并行）

将模型的不同层分布到不同GPU上，形成流水线。

适用于：跨节点多GPU场景

```text
GPU0: Layer 1-10  --->  GPU1: Layer 11-20  --->  GPU2: Layer 21-30
          |                      |                        |
       输入                   中间激活                   输出
```

- 通信量：仅为中间激活（较小）
- 利用率问题：存在"气泡"（pipeline bubble）
- 缓解：Micro-batching（将batch拆分为micro-batch）

#### 3. Expert Parallel for MoE（MoE专家并行）

针对Mixture-of-Experts模型的专用并行策略。

- 每个Expert放置在不同GPU
- 路由（Router）分配token给对应的Expert
- 需要all-to-all通信

```text
Router: 输入token被路由到不同Expert
         |
  Expert A (GPU0)  ---  token子集A
  Expert B (GPU1)  ---  token子集B
  Expert C (GPU2)  ---  token子集C
  Expert D (GPU3)  ---  token子集D
         |
         合并输出
```

典型模型：Mixtral 8x7B、DeepSeek-V2/V3、Qwen-MoE

#### 1. GPU实例选型

| 需求场景 | 推荐GPU | 显存需求 | 参考成本 |
|---------|--------|---------|---------|
| 7B模型推理 | A10G / RTX 4090 | 16-24GB | 低 |

选型要素：

- 计算能力：TFLOPS（FP16/BF16/FP8）
- 显存带宽：HBM带宽影响解码速度
- 互联带宽：NVLink vs PCIe

#### 2. 负载均衡

请求路由策略：
- Round Robin：简单均匀分配
- Least Connections：根据当前连接数分配
- Token Aware：根据每个实例剩余容量（可用token slots）分配


#### 4. 冷启动优化

问题：模型加载到GPU需要数分钟，导致弹性扩缩容延迟高。

方案：
1. 预加载（Model Warm Pool）
   - 提前部署N个副本并保持空闲
   - 收到请求快速切换

2. 模型分片加载
   - 先加载模型部分层开始服务
   - 后台继续加载剩余层

3. 容器镜像优化
   - 使用模型缓存（model cache volume）
   - 分层加载（先加载权重、再加载代码）

4. Serverless推理
   - AWS SageMaker Serverless Inference
   - 分钟级冷启动（优化后可降至10秒内）

### 九、面试常见问题

#### Q1: 如何优化LLM推理延迟？

回答框架：
1. **模型层面**：量化（INT4/AWQ）、模型蒸馏、MoE激活稀疏
2. **推理引擎**：使用vLLM/TensorRT-LLM、启用Flash Attention
3. **KV Cache优化**：PagedAttention、Prefix Caching、KV Cache量化
4. **并行策略**：Tensor Parallel、增大batch size
5. **部署优化**：Speculative Decoding（投机解码）、Prompt压缩

#### Q2: 设计一个支持100并发的LLM推理服务？

回答框架：
1. **选型**：vLLM + Ray Serve / KServe
2. **GPU配置**：
   - 每个模型副本使用A100 80G
   - Tensor Parallel=2（如果模型较大）
3. **并发处理**：
   - Continuous Batching自动处理并发
   - 配置max-num-seqs=256
4. **弹性扩展**：
   - 部署5-10个副本（根据需求）
   - 自动扩缩容
5. **负载均衡**：
   - Token-aware调度
   - 请求队列（Redis + Celery）
6. **优化**：
   - 启用Prefix Caching
   - 响应流式输出（SSE/Streaming）

#### Q3: 如何选择量化方案？

决策树：
```
模型 > 7B 且需要最高精度？
  └─ 是 -> AWQ 4bit（精度最高）
  └─ 否 -> 继续
需要CPU推理？
  └─ 是 -> GGUF Q4_K_M/Q5_K_M
  └─ 否 -> 继续
需要最大速度？
  └─ 是 -> GPTQ/TensorRT-LLM INT8
  └─ 否 -> AWQ 4bit（最佳平衡）
边缘设备？
  └─ 是 -> TFLite INT8 / CoreML FP16
```

#### Q4: 如何优化Serverless推理冷启动？

方案：
1. **模型预加载池（Warm Pool）**
   - 维护N个预热实例
   - 请求涌入时快速切换

2. **模型存储优化**
   - 使用NVMe本地SSD缓存
   - 分布式模型缓存（如S3FS-FUSE + Caching）

3. **模型分片加载**
   - 先加载前几层开始服务
   - 异步加载剩余层

4. **容器优化**
   - 使用Safetensors加速权重加载
   - 优化Docker镜像层（基础镜像预装CUDA库）

5. **框架优化**
   - vLLM的模型加载比HuggingFace Transformers快
   - 使用编译后的Engine（TensorRT-LLM）

# AI应用开发框架面试知识

## 一、概述

AI应用开发框架是当前大模型落地工程化的核心基础设施。2025-2026年，主流框架已从单纯的LLM调用封装演进为完整的AI应用开发平台，涵盖模型编排、状态管理、检索增强、多智能体协作等能力。面试中，候选人需要理解各框架的核心设计理念、适用场景、关键差异以及最新演进方向。

## 二、知识点详解

### 1. LangChain 核心概念

#### 1.1 Chain
Chain是LangChain最原始的设计抽象，代表一系列组件的有序组合。典型如 LLMChain（Prompt + Model + OutputParser）、SequentialChain（串联多个Chain）、RouterChain（条件路由）。但自 v0.2 起，Chain 已被标记为 Legacy，推荐使用 LCEL。
```python
# Legacy Chain 方式
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate
from langchain.llms import OpenAI

chain = LLMChain(
    llm=OpenAI(),
    prompt=PromptTemplate.from_template("介绍{topic}")
)
result = chain.run(topic="LangChain")
```

#### 1.2 Runnable（LangChain v0.2+核心接口）
Runnable 是 LangChain 统一执行接口，任何可执行组件都实现 `Runnable` 协议（`invoke`、`batch`、`stream`、`ainvoke` 等）。核心子类包括 RunnableSequence、RunnableParallel、RunnableLambda、RunnablePassthrough、RunnableBranch 等。
```python
from langchain_core.runnables import RunnableLambda, RunnablePassthrough

runnable = (
    {"topic": RunnablePassthrough()}
    | RunnableLambda(lambda x: f"介绍{x['topic']}")
)
result = runnable.invoke("LangChain")
```

#### 1.3 LCEL（LangChain Expression Language）
LCEL 是 LangChain 的声明式编排语言，使用 `|` 管道操作符组合组件。LCEL 自动支持流式、异步、批处理、并行和重试。
```python
from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

prompt = ChatPromptTemplate.from_template("用{language}写一段代码：{topic}")
model = ChatOpenAI(model="gpt-4o")
output_parser = StrOutputParser()

chain = prompt | model | output_parser
result = chain.invoke({"language": "Python", "topic": "二分查找"})
```

**LCEL 优势**：类型安全、自动支持流式/异步、编译期优化执行图、简洁可读。

#### 1.4 Tool
Tool 是 LLM 可以调用的外部能力接口，包含名称、描述、参数 schem a 和执行逻辑。Tool 可通过 @tool 装饰器或 BaseTool 子类定义。
```python
from langchain_core.tools import tool
import requests

@tool
def get_weather(city: str) -> str:
    """获取指定城市的天气信息"""
    response = requests.get(f"https://api.weather.com/{city}")
    return response.text
```

#### 1.5 Agent
Agent 是使用 LLM 作为推理引擎，动态决定调用哪些 Tool 以及如何组合的执行器。LangChain 提供了多种 Agent 类型，ReAct Agent 是最常用的。
```python
from langchain.agents import create_react_agent, AgentExecutor
from langchain_openai import ChatOpenAI

agent = create_react_agent(
    llm=ChatOpenAI(model="gpt-4o"),
    tools=[get_weather, search_tool],
    prompt=react_prompt
)
agent_executor = AgentExecutor(agent=agent, tools=[get_weather, search_tool])
result = agent_executor.invoke({"input": "北京明天天气如何？"})
```

2025-2026年趋势：LangChain 已全面拥抱 LangGraph 作为 Agent 的底层执行引擎，AgentExecutor 内部本质是一个 LangGraph StateGraph。

### 2. LangGraph（状态驱动的Agent框架）

LangGraph 是 LangChain 团队开发的**有状态图执行引擎**，专为Agent和多轮对话场景设计。

#### 2.1 StateGraph
StateGraph 的核心概念：节点（Node）、边（Edge）、状态（State）。节点执行逻辑更新状态，边决定执行流向。
```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated, List
import operator

class AgentState(TypedDict):
    messages: Annotated[List, operator.add]
    next_step: str

def node_a(state: AgentState):
    return {"messages": ["执行步骤A"], "next_step": "b"}

def node_b(state: AgentState):
    return {"messages": ["执行步骤B"], "next_step": END}

graph = StateGraph(AgentState)
graph.add_node("a", node_a)
graph.add_node("b", node_b)
graph.set_entry_point("a")
graph.add_conditional_edges("a", lambda s: s["next_step"], {"b": "b"})
graph.add_edge("b", END)
app = graph.compile()
```

#### 2.2 AgentExecutor（LangGraph版）
LangGraph 提供了 `create_react_agent` 函数替代传统 AgentExecutor，内部封装了 ReAct 的循环逻辑——思考(Thought) → 行动(Action) → 观察(Observation) → 循环直到结束。
```python
from langgraph.prebuilt import create_react_agent

agent = create_react_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[get_weather, search_tool]
)
events = agent.stream({"messages": [("human", "北京明天天气如何？")]})
for event in events:
    print(event)
```

#### 2.3 Human-in-the-Loop
LangGraph 原生支持人类介入的三种模式：
- **中断(Interrupt)**：在指定节点暂停，等待人工输入
- **审批(Approval)**：Agent 执行关键操作前需要人类确认
- **编辑(Editing)**：支持人类修改 Agent 的下一步计划

```python
from langgraph.checkpoint.memory import MemorySaver

# 在关键节点前设置中断
graph = graph.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["execute_payment"]  # 执行支付前暂停
)
```

#### 2.4 LangGraph 2025-2026新特性
- **LangGraph Cloud**：托管的Agent部署平台，支持持久化状态、长时间运行的Agent
- **LangGraph Studio**：可视化Graph调试工具
- **Subgraph**：子图复用机制，支持图嵌套
- **Persistence Layer**：支持 PostgreSQL、Redis 等持久化后端

### 3. LlamaIndex

LlamaIndex 聚焦于**数据索引与检索增强生成(RAG)**场景，提供从数据接入到查询的完整数据管线。

#### 3.1 Index结构
LlamaIndex 支持多种索引类型：
- **VectorStoreIndex**：向量存储索引，语义搜索
- **SummaryIndex**：摘要索引，存储文档摘要
- **KeywordTableIndex**：关键词索引，BM25检索
- **TreeIndex**：树形索引，递归总结
- **PropertyGraphIndex**：属性图索引，2024年新增

```python
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

documents = SimpleDirectoryReader("./data").load_data()
index = VectorStoreIndex.from_documents(documents)
query_engine = index.as_query_engine()
```

#### 3.2 Retriever
Retriever 负责从 Index 中检索相关信息。LlamaIndex 提供了丰富的检索策略：
```python
from llama_index.core.retrievers import VectorIndexRetriever
from llama_index.core.query_engine import RetrieverQueryEngine

retriever = VectorIndexRetriever(
    index=index,
    similarity_top_k=5
)

# 高级检索：混合检索 + 重排序
from llama_index.core.retrievers import RouterRetriever
from llama_index.core.retrievers import BM25Retriever
```

#### 3.3 Query Engine
Query Engine 是统一查询接口，内部封装了检索+合成的完整逻辑。
```python
# 基础查询
response = query_engine.query("文档的主要内容是什么？")

# 多步查询——逐步细化
from llama_index.core.query_engine import MultiStepQueryEngine
```

#### 3.4 Data Ingestion
2024-2025年，LlamaIndex 推出了 `IngestionPipeline`，支持数据清洗、分块、嵌入、存储的一站式处理。
```python
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter
from llama_index.embeddings.openai import OpenAIEmbedding

pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=1024, chunk_overlap=200),
        OpenAIEmbedding(),
    ]
)
nodes = pipeline.run(documents=documents)
```

#### 3.5 Agent能力
LlamaIndex 在 2025 年强化了Agent能力，支持 Function Calling Agent、ReAct Agent 以及多Agent编排。

### 4. Semantic Kernel（微软）

Semantic Kernel（SK）是微软推出的轻量级AI编排框架，深度集成Azure生态和.NET/C#生态，同时支持Python。

#### 4.1 Plugin
Plugin 是 SK 的核心能力单元，分为 Native Plugin（原生代码）和 Semantic Plugin（定义在prompt文件中的语义函数）。
```python
from semantic_kernel import Kernel
from semantic_kernel.plugin_definition import kernel_function

class WeatherPlugin:
    @kernel_function(description="获取天气")
    def get_weather(self, city: str) -> str:
        return f"{city}的天气晴朗"

kernel = Kernel()
kernel.add_plugin(WeatherPlugin(), plugin_name="weather")
```

#### 4.2 Planner
Planner 是 SK 的任务规划引擎，基于LLM自动生成执行计划。SK v1.0+ 引入了 HandlebarsPlanner 和 FunctionCallingStepwisePlanner。
```python
from semantic_kernel.planners import FunctionCallingStepwisePlanner

planner = FunctionCallingStepwisePlanner(
    kernel=kernel,
    service_id="gpt-4o"
)
result = await planner.invoke_async("帮我查询北京的天气并发送邮件")
```

#### 4.3 Memory
SK 的 Memory 模块支持语义记忆和向量存储，类似外部知识库。
```python
from semantic_kernel.memory import SemanticTextMemory
from semantic_kernel.connectors.memory.azure_cognitive_search import AzureCognitiveSearchMemoryStore
```

#### 4.4 2025-2026年进展
- **SK v1.x 稳定版**：API 趋于稳定，Python版与C#版功能对齐
- **Auto Function Calling**：原生支持 OpenAI/Anthropic 的工具调用
- **Agent Framework**：SK Agent 子框架，支持多Agent
- **Copilot Stack**：与 Microsoft 365 Copilot 深度集成

### 5. Haystack（deepset）

Haystack 是专注于搜索和RAG的开源框架，强调模块化和生产级可用性。

#### 5.1 Pipeline
Haystack 的 Pipeline 是有向无环图（DAG），节点间通过连接传递数据。
```python
from haystack import Pipeline
from haystack.components.retrievers import InMemoryEmbeddingRetriever
from haystack.components.generators import OpenAIGenerator

pipeline = Pipeline()
pipeline.add_component("retriever", InMemoryEmbeddingRetriever(...))
pipeline.add_component("generator", OpenAIGenerator(model="gpt-4o"))
pipeline.connect("retriever.documents", "generator.documents")
result = pipeline.run({"retriever": {"query": "什么是RAG？"}})
```

#### 5.2 Document Store
支持多种文档存储后端：InMemoryDocumentStore、ElasticsearchDocumentStore、WeaviateDocumentStore、PineconeDocumentStore 等。
```python
from haystack.document_stores.in_memory import InMemoryDocumentStore
document_store = InMemoryDocumentStore(embedding_similarity_function="cosine")
```

#### 5.3 Reader-Generator
Haystack v2.x（2024+）统一使用 Generator 概念替代原来的 Reader：
```python
from haystack.components.generators import OpenAIGenerator, AnthropicGenerator, HuggingFaceLocalGenerator
```

#### 5.4 2025-2026特色
- **Haystack 2.x**：完全重写的组件化架构，异步原生支持
- **Deepset Cloud**：托管的 RAG 平台
- **Evaluation Pipeline**：内置RAG评估管线
- **Agent 支持**：引入工具调用和Agent组件

### 6. DSPy框架

DSPy（Declarative Self-improving Python）是斯坦福大学开发的**声明式LLM编程框架**，核心理念：将Prompt Engineering转化为编程问题，通过编译器自动优化prompt。

#### 6.1 声明式LLM编程
DSPy 使用 Module 抽象替代手写Prompt：
```python
import dspy
from dspy import ChainOfThought, Predict

class AnswerQuestion(dspy.Module):
    def __init__(self):
        super().__init__()
        self.generate_answer = ChainOfThought("context, question -> answer")
    
    def forward(self, context, question):
        return self.generate_answer(context=context, question=question)
```

#### 6.2 Signature
Signature 是 DSPy 的核心抽象，定义输入和输出字段的声明式规范：
```python
class ExtractInfo(dspy.Signature):
    """从文本中提取结构化信息"""
    text: str = dspy.InputField(desc="原始文本")
    name: str = dspy.OutputField(desc="提取的名称")
    age: int = dspy.OutputField(desc="提取的年龄")
    occupation: str = dspy.OutputField(desc="职业信息")
```

#### 6.3 Optimizer（编译器）
DSPy 的自动优化器（原名 Teleprompter）可以自动优化prompt：
- **LabeledFewShot**：从标注数据中自动选择示例
- **BootstrapFewShot**：通过LLM自举生成示例
- **BootstrapFewShotWithRandomSearch**：带随机搜索的优化
- **MIPROv2**：多指令提议优化（2024年提出，2025年成为推荐方案）
- **COPRO**：坐标下降优化

```python
from dspy.teleprompt import MIPROv2

optimizer = MIPROv2(prompt_model=dspy.LM("gpt-4o"))
compiled_program = optimizer.compile(
    AnswerQuestion(),
    trainset=trainset,
    num_trials=10
)
```

#### 6.4 2025-2026亮点
- **DSPy v2.5+**：支持多模态输入（图像、音频）
- **DSPy Assistant**：交互式优化助手
- **跨框架兼容**：可生成 LCEL-compatible 的优化结果
- **评估集成**：内置评估框架，支持RAGAS等评估指标

## 三、面试常见问题（5个）

### Q1: LCEL 与传统 Chain 模式有什么区别？为什么 LangChain 要迁移到 LCEL？

**答**：
1. **执行模型**：传统Chain是硬编码的执行顺序，LCEL 使用惰性求值编译为优化执行图
2. **流式支持**：LCEL 自动支持流式输出，传统Chain需要手动实现
3. **并行执行**：LCEL 中 RunnableParallel 自动并行，传统Chain需额外设计
4. **类型安全**：LCEL 编译时检查类型兼容性
5. **可观测性**：LCEL 自动生成执行追踪（LangSmith集成）
6. **组件复用**：LCEL 的 Runnable 接口统一，可自由组合

迁移原因：传统Chain难以扩展，不支持流式、并行、动态路由等现代AI应用需求。

### Q2: LangGraph 的 StateGraph 与传统 DAG 框架的核心区别是什么？

**答**：
1. **状态中心**：StateGraph 维护全局状态（State），节点只关心状态的读取和更新，类似于 Redux 的状态管理模式
2. **条件边**：支持基于当前状态动态决定执行路径，而非静态DAG
3. **循环支持**：天然支持循环（Agent的 Thought-Action-Observation 循环），传统DAG多为有向无环图
4. **中断与恢复**：原生支持在任意节点中断，保存checkpoint，后续恢复执行
5. **并发节点**：多个节点可以并行修改同一状态（需定义归约函数 reducer）

### Q3: 如何实现 AI 框架的线程安全？在多用户并发场景下需要注意什么？

**答**：
- **LangChain/LangGraph**：Runnable 默认是无状态的，但 Callback 和 LangGraph 的 State 需要关注线程安全。使用独立的 Runnable 实例或 Deep Copy。LangGraph 的 Checkpointer（MemorySaver）不是线程安全的，需要改用 PostgreSQL/Redis 后端。
- **LlamaIndex**：ServiceContext 不是线程安全的，高并发下应每个请求创建独立的 ServiceContext。
- **通用方案**：
  - 无状态化设计：尽可能让组件无状态
  - 连接池：向量数据库/LLM客户端使用连接池
  - 异步处理：使用 `ainvoke`/`arun` 等异步接口，配合 asyncio 协程
  - 每个请求独立实例：在 FastAPI 等框架中为每个请求创建独立的 Chain/Agent 实例

### Q4: AI框架中的 Streaming 是如何实现的？不同框架的实现差异？

**答**：
- **LangChain LCEL**：Runnable 接口自带 `stream()` 方法，LCEL 自动在组件间逐 token 传递流式数据。核心是 RunnableGenerator 的 yield 实现。
- **LangGraph**：支持 `stream_mode=["values", "updates"]`，可以流式输出节点状态变化，也可以逐 token 流式输出 LLM 的生成。
- **LlamaIndex**：`StreamingQueryEngine` 包装了流式响应，需要传 `streaming=True` 参数。
- **Semantic Kernel**：支持 `StreamingChatCompletion` 接口。
- **通用原理**：底层依赖 LLM API 的 SSE（Server-Sent Events）实现，框架层通过 Generator/Python Async Generator 逐段传递给上层调用者。

### Q5: 如何选择合适的框架构建一个企业级 RAG 应用？

**答**：
1. **需求分析**：
   - 文档量级：<10K文档 → 任何框架均可；10K-1M → LlamaIndex/Haystack；>1M → 需自定义 + 高性能向量库
   - 检索复杂度：简单检索 → 任一框架均可；多步检索/多轮对话 → LangGraph
   - 部署环境：Azure/C# → Semantic Kernel；云原生 → LangGraph Cloud/Deepset Cloud
   
2. **推荐方案**：
   - 文档问答（简单RAG）：LlamaIndex + 向量数据库
   - 复杂Agent + RAG：LangChain/LangGraph + LlamaIndex（两者可以混用）
   - 搜索型产品：Haystack + Elasticsearch
   - 需要极致Prompt优化：DSPy优化后再部署到 LangChain/LlamaIndex
   
3. **组件可替换性**：核心原则——框架只是脚手架，关键在正确选择 Embedding 模型、Chunking 策略、检索方式和 LLM。
